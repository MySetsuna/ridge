# Ridge 多智能体协同 · 用户使用手册

> 适用版本：develop（≥ 2026-06-20「底座化瘦身」之后）。需重新构建/安装才能使用本手册描述的能力。
> 设计依据见 `docs/superpowers/specs/2026-06-20-team-agent-upgrade-plan-design.md`。

---

## 这是什么

Ridge 内置一套**端侧、本地优先**的多智能体协同能力——你可以在同一个 Ridge 窗口里让多个 AI 智能体（Agent）**分屏并肩工作**，而你坐在中间，**看清状态、拦住危险、断开死循环、按需接入任意 agent**。

它的定位很克制：**Ridge 是"一个人安全地、就地指挥一群可见 panes 的终端底座"，不是一个替你做决策的 AI 编排平台。** 协调由**你**来做（agent 就是你眼前的分屏），Ridge 专心占住 agent 占不住的东西：本地机器、安全闸、你的注意力、共享可见的工作区。

围绕四件事设计：

| 能力 | 一句话 |
| --- | --- |
| 👁️ **看得见**（状态面板 / Agent Center） | 侧栏面板，实时显示团队成员与异常告警 |
| 🛡️ **拦得住**（安全审批 / HITL） | 智能体要执行危险命令时，弹窗让你**批准 / 拒绝 / 改写** |
| ⚡ **断得了**（循环熔断） | 某个智能体陷入「改→测→报错」死循环时，自动中断并红色告警 |
| 🔌 **接得上**（内置 MCP server） | 标准 MCP 协议端点，让 MCP-原生的智能体（如 Claude Code）直接挂进来 |

> **设计说明**：早期版本曾内置一层"AI 之间自治协同"（智能体互发 TML 消息、自动竞选 Leader、广播抢单、性格分派）。2026-06-20 起这一层**已整体退场**——它把干净的"人协调可见分屏"模型推向脆弱、不可解释、且会被更强基座淘汰的方向，违背 Ridge 的硬核极简。保留下来的，是上面四样**给人用的**能力。

---

## 核心概念速览

- **Agent（智能体）= 一个分屏（pane）**：每个跑着 AI 的终端分屏就是一名队员。
- **角色**：Leader / Worker / Observer。Leader 只由**人类静态钦定**（不再 AI 自动竞选）。
- **风险分级 L0 / L1 / L2**：每条命令按危险度分三档（只读 / 写工作区 / 危险），是安全审批的依据。

---

## 1. 状态面板（Agent Center）

**在哪**：左侧栏的「**智能体**」Tab。看不到时，先确认侧栏已展开、且当前有打开的工作区。

瘦身后只保留两块**给人看的**信息：

### 成员（Roster）
列出所有队员：👑 皇冠 = Leader（人类钦定时才有）；每个名字右边一个**状态点**：绿色脉冲 = 工作中，灰 = 空闲，暗灰 = 已失联。右上角数字 = 人数。没有队员时显示「暂无成员」。

### 异常（熔断告警，仅异常时出现）
当某个队员被判定陷入死循环并熔断时，这里会**置顶一条红色告警**（谁、为什么），右上角有「清除」按钮。平时无异常时**完全不显示**，不打扰。

> 顶部还有一个「审批 开/关」快捷开关（HITL），等价于设置面板里的安全审批开关。

---

## 2. 安全审批网关（HITL · Human-in-the-Loop）

这是 Ridge 的**招牌护城河**：自治 agent 直接碰你的真实机器，这一层是你的**最后一道闸**。

**开启后**，任何智能体试图提交一条被判为 **L2（危险）** 的命令时，Ridge 会**挂起那条命令、不让它执行**，弹出一个居中的审批模态：

```
┌─────────────────────────────────────────────┐
│ 🛡️ 需要你的授权                         [L2]  │
│    来自 pane#2 (claude-code)                  │
│ ───────────────────────────────────────────  │
│ 递归/批量删除                                  │
│ ┌─────────────────────────────────────────┐ │
│ │ rm -rf /tmp/build-cache --no-preserve-root│ │
│ └─────────────────────────────────────────┘ │
│  ✕ 拒绝              ✏ 修改        ✓ 批准      │
└─────────────────────────────────────────────┘
```

三个选择：**✓ 批准**（放行）/ **✕ 拒绝**（阻断，向智能体返回「被授权策略拦截」）/ **✏ 修改**（改写命令后再执行）。

**特性**：
- 多条待审批会**排队**，模态右上角 `+N` 表示后面还有几条。
- 超时未裁决（默认 120 秒）会**自动拒绝**（fail-closed，宁可挡错不可放错）。
- **只拦 L2 危险命令**；L0/L1 直接放行，不打扰你。
- **独立安全闸，不可被整体关**：一旦你开启 HITL，它的生效**不受「智能体协同」总开关左右**——关掉总开关（隐藏面板）**不会**静默撤销你的安全闸。对标 Claude Code「hooks 即便 yolo 也触发」。

> ⚠️ **默认关闭**。开关在设置面板「智能体」分区，或 Agent Center 顶部快捷开关。关闭时智能体命令照常执行（行为与加这套系统之前完全一致）。

## 3. 风险分级（L0 / L1 / L2）

| 档位 | 含义 | 例子 |
| --- | --- | --- |
| **L0** 只读 | 不改任何东西 | `ls -la`、`git status`、`cat x`、`grep ...` |
| **L1** 工作区写 | 在工作区内写文件 | `echo hi > out.txt`、`npm install`、改文件 |
| **L2** 危险 | 可能造成不可逆/外部影响 | `rm -rf ...`、`git push`、`curl ... | sh`、`sudo ...` |

分级器**抗绕过**：先归一化多余空格（`git   push` 照样判 L2）、识别管道灌 shell（`... | bash`）、重定向覆盖等花招。

---

## 4. 循环熔断

AI 智能体偶尔会卡进「改代码 → 跑测试 → 同样报错 → 再改 → 还是同样报错」的死循环。循环熔断是给每个 Worker 装的**保险丝**。

**怎么触发**：当一个队员**连续 3 次**上报**高度相似的失败**（同一错误特征），系统判定逻辑死锁，于是：① 向那个分屏发 `Ctrl+C` 强制打断；② 在 Agent Center **置顶一条红色「异常」告警**。

你接管那个队员（重新给指令、换思路）后点「清除」消掉告警。

> 熔断**随队工作**（无需手动开），但只有当队员通过协作 API 上报带「分屏 + 失败特征」的进度时才会计数。阈值（3 次）内置。它是一个**哑保险丝**——只负责断+告警，不替你重新规划。

## 5. 把任务发给某个分屏（delegate / report）

队员之间有一组内部 HTTP API（`delegate-task` / `report-progress` / `team-profile`），由 agent 运行时调用：

- **派活（delegate-task）**：把一个目标发给某个分屏，注入任务提示词唤醒它、标记「工作中」，返回一张任务票。**由人/调用方显式发起**（不再有 AI Leader 自动派活，也没有"广播抢单"）。
- **汇报（report-progress）**：把阶段性结果回报；带「分屏 + 失败特征」时同时喂**循环熔断**计数。
- **查兵马（team-profile）**：拉取花名册。

你通常不用直接碰它们。

## 6. 内置 MCP server（高级 / 给开发者）

Ridge 自带一个标准 **MCP（Model Context Protocol）** 服务端，让 MCP-原生的智能体直接挂进来用 Ridge 的能力——这是 Ridge"做开放底座、不做大脑"战略的化身。

- **端点**：`ws://<teammate-host>/api/v1/mcp/ws`（带 bearer token 鉴权；地址/令牌通过分屏环境变量 `RIDGE_TEAMMATE_*` 注入）。
- **协议**：JSON-RPC 2.0，支持 `initialize` / `tools/list` / `tools/call` / `resources/read`。
- **工具**：`ridge_send_to_teammate`、`ridge_delegate_task` 等。
- **资源**：`ridge://workspace/active-panes`（活动工作区花名册）等。

普通用户不用关心这一层。

## 7. 开关一览

| 能力 | 默认 | 开关 | 说明 |
| --- | --- | --- | --- |
| **安全审批（HITL）** | 关 | 设置面板「智能体」分区 / Agent Center 顶部 | 开后对 L2 命令弹审批模态；**独立于总开关**，不可被整体关 |
| **循环熔断** | 开（随队工作） | 无 | 队员上报失败达阈值即触发 |
| **状态面板** | 开（常驻侧栏） | 无 | 无团队时显示空态，不打扰 |
| **智能体协同总开关** | 开 | 设置面板「智能体」分区 | **仅控制 UI 露出**（Tab / pane 入口）；不影响安全闸 |

> HITL 之所以默认关，是因为它触及命令执行关键路径——先确保对现有使用**零影响**，待你确认要用再开。关闭时 send-keys 字节级零变化。

## 8. 常见问题（FAQ）

**Q：我看不到「智能体」面板？**
A：① 确认用的是包含本能力的新版本（需重新构建/安装）；② 确认左侧栏已展开、且当前有打开的工作区。

**Q：花名册一直为空 / 显示「暂无成员」？**
A：只有当智能体通过 Ridge 的 teammate 机制**注册**到某个分屏后才会进花名册。你自己手动开的普通终端不算队员。

**Q：我没看到危险命令的审批弹窗，它直接跑了？**
A：安全审批**默认关闭**，需先在设置里开。另外它**只拦 L2**，且只拦智能体经协作通道提交的命令——你自己手敲的命令不走这条审批闸。

**Q：我关掉了「智能体协同」总开关，安全审批还在吗？**
A：**在**。HITL 是独立安全闸——总开关只隐藏 UI，不会撤销你已开启的审批闸。

**Q：熔断会误伤吗？**
A：只有「连续 3 次**高度相似**的失败」才触发；单次失败、或失败特征不同，都不会熔断。触发后是 `Ctrl+C` 打断 + 告警，不杀分屏。

**Q：这套东西会拖慢我平时用终端吗？**
A：不会。HITL 默认关时是**字节级零变化**；状态面板无团队时零渲染。

## 9. 当前状态与前提（请先读）

- ✅ **已实现并验证**：四块"给人用的"能力（状态面板 / HITL / 熔断 / MCP）的纯核心 + 前端（`pnpm check` 0/0、ridge-core 单测绿），后端接线编译验证通过。
- ⚙️ **使用前提**：需要**重新构建/安装** develop 上的最新版本；本能力尚未进入正式 Release。后端瘦身改动**待真机 e2e**。
- 🧹 **已退场（别再找）**：智能体互发 TML 消息、协作审计、AI 自动竞选 Leader、广播抢单、性格分派、拓扑 DAG。这些"给 AI 之间自治用的"脚手架已在 2026-06-20 底座化瘦身中删除。
- 🔭 **观望（按真实使用再决定）**：事件持久化/历史、技能复利、并行竞速判优、worktree 隔离档、多厂商接入抽象。现在一律不做——见设计文档 §5。
