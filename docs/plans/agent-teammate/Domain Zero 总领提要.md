在智能体团队协同工作流设计对话中，通过梳理我的意图，可以形成以下方向性的提要。这些意图展示了我希望将 Ridge 从一个单点终端工具进化为**“端侧多智能体协同自治系统（Local Multi-Agent Orchestration Engine）”**的宏大愿景。

### 1. 核心愿景：构建“智能体物理社交与协作网络”
我希望 Ridge 不仅仅是一个 AI 辅助工具，而是演变为一个**具备物理运行环境（PTY、文件系统、CLI）的 Agent 社交空间**。
*   **从线性到网状**：将模式从“1个用户 ↔ 1个 Agent”升级为“1个用户 ↔ 1个智能体团队 ↔ 共享工作区终端群”的网状协作模式。
*   **端侧自治**：强调在本地通过 **Teammate API** 总线或共享 PTY 进行高频、实时沟通，避免云端 API 的高延迟。

### 2. 执行哲学：“去中心化”与“动态加冕”
我提出了一个极其精妙的架构思想，即 Ridge 不直接作为 LLM 客户端管理 Key，而是作为**智能体的物理操作系统**。
*   **利用现有生态**：直接调用用户本地已配置好的 Claude Code、OpenCode 等 CLI 工具作为智能体，实现“零配置、开箱即用”。
*   **动态加冕机制**：用户可以随时将某个 Pane（分屏）中的普通 Agent 标记为 **Team Leader（皇冠标记）**，赋予其指挥权，而非依赖死板的后台黑盒进程。
*   **自主切换**：被加冕的 Leader 可以根据任务复杂度，通过指令自主切换模型（如从 Haiku 切换到 Sonnet）或调整性格。

### 3. 协作机制：物理召回与“并肩作战”
我强调多智能体协作必须是**“肉眼可见”**的，从而提供极致的极客体验。
*   **Summon（召唤）功能**：后台静默工作的 Worker Agent 在需要时，由 Leader 或用户“召唤”到前台，通过**递归分屏**挂载到当前工作区，展示实时输出。
*   **无感收养 PTY**：召唤过程应是数据驱动的，Worker 在后台积攒的日志会在分裂的一瞬间刷在屏幕上，实现“隐身”到“显身”的无缝切换。
*   **异构协同**：根据任务目标，自动评估 Agent 的**能力矩阵（Capability Matrix）**和**性格倾向**来分配工作（如严谨型负责测试，激进型负责快速重构）。

### 4. 通信架构：双层驱动的“基本盘 + 强化包”
我坚持系统应具备极高的兼容性，将通信分为两个层次：
*   **物理骨架层（基本盘）**：利用现有的 **tmux shim 拦截** 和 **PTY 物理代打** 模式。即使 Agent 不懂任何协议，也能通过在终端“强灌文本”实现喊话和派活。
*   **MCP 增强层（强化项）**：将 **MCP (Model Context Protocol)** 作为可选插件。开启后，Agent 可通过结构化 JSON-RPC 实现静默通信、秒传大资产（如 AST 树）和异步任务通知，不污染终端屏幕。

### 5. 用户体验：上帝视角与安全审计
我意图让用户始终保持对“赛博团队”的最高控制权。
*   **Agent Center 侧边栏**：作为一个独立的“数字指挥部”，展示团队拓扑图、各 Agent 状态、性格配置及 Summon 入口。
*   **人类中间审批（HITL）**：针对删除文件、提权或 git push 等高危操作，建立**拦截网关**，由用户在前端 UI 进行确认或修改指令。
*   **异常熔断**：当 Agent 陷入死循环或发生文件写入冲突时，系统能自动检测并触发物理级中断，交由人类裁决。
