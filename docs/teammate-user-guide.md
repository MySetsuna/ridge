# Ridge 多智能体协同 · 用户使用手册

> 适用版本：develop（≥ 本次「Domain Zero 端侧多智能体协同」落地后）。需重新构建/安装才能使用本手册描述的能力。

---

## 这是什么

Ridge 不再只是一个「你一个人对着一个终端」的工具。它现在内置了一套**端侧多智能体协同自治系统**——多个 AI 智能体（Agent）可以在同一个 Ridge 窗口里**分屏协作**：一个当**指挥官（Leader）**统筹，其余当**工人（Worker）**干活；它们之间能互相派活、汇报、求助，而你坐在中间，通过一个**数字指挥部**看清全局、并在关键时刻**一键拍板**。

这套系统围绕四件事设计：

| 能力 | 一句话 |
| --- | --- |
| 👁️ **看得见**（数字指挥部 / Agent Center） | 一个侧栏面板，实时显示团队成员、当前目标、协作日志、异常告警 |
| 🛡️ **拦得住**（安全审批 / HITL） | 智能体要执行危险命令时，弹窗让你**批准 / 拒绝 / 改写**，而不是直接跑 |
| ⚡ **断得了**（循环熔断） | 某个智能体陷入「改→测→报错」死循环时，自动中断它并红色告警，喊你接管 |
| 🔌 **接得上**（内置 MCP server） | 标准 MCP 协议端点，让 MCP-原生的智能体（如 Claude Code）直接挂进来协作 |

---

## 核心概念速览

- **Agent（智能体）= 一个分屏（pane）**：每个跑着 AI 的终端分屏就是一名队员。
- **Leader / Worker / Observer**：角色。Leader 由「能力画像」自动竞选产生（语言技能、上下文窗口、细致度加权），统筹分派；Worker 执行；Observer 旁观。
- **风险分级 L0 / L1 / L2**：每条命令按危险度分三档（只读 / 写工作区 / 危险），是安全审批的依据。
- **TML（Teammate Markup Language）**：智能体之间互相「说话/派活/汇报」用的内部消息格式，你不用懂它——指挥部会把它**翻译成人话**给你看。

## 1. 数字指挥部（Agent Center）

**在哪**：左侧栏底部的「**智能体指挥部**」面板（和「全局状态」并列的全局插件区）。如果看不到，先确认侧栏已展开、且当前有打开的工作区。

它分四块，自上而下：

### 当前目标
显示团队当前在干嘛，例如「指挥官 Claude 统领 3 名成员」。没有活跃团队时显示「暂无活跃团队」。

### 团队花名册
列出所有队员：
- 👑 **皇冠**标记的是 **Leader**。
- 每个名字右边有个**状态点**：绿色脉冲 = 工作中（Working），灰色 = 空闲（Idle），暗灰 = 已失联（Disappeared）。
- 右上角数字 = 团队人数。
- 没有队员时显示「还没有智能体加入」。

### 协作审计
把智能体之间晦涩的内部通信**降维成人话**，倒序滚动，例如：
- `Claude 给 Hermes 派活：为缓存层补单元测试`
- `Hermes 向 Claude 汇报：45 个用例全部 PASS`
- `Claude 对 Hermes 说话`

> 这一栏只有在**「TML 流净化」开关打开**（见 §7）且队员之间真的在用 TML 通信时才会有内容。

### 熔断告警（仅异常时出现）
当某个队员被判定陷入死循环并熔断时，这里会**置顶一条红色告警**：
- 🔴 `Hermes 已熔断 · 同一文件 改→测→报错 ×3 (E0277 类型不匹配)`
- 右上角有「**清除**」按钮，你确认接管后点掉即可。

平时无异常时这一栏**完全不显示**，不打扰。

## 2. 安全审批网关（HITL · Human-in-the-Loop）

智能体自动跑命令很方便，但「自动 `rm -rf`」很可怕。安全审批网关就是你的**最后一道闸**。

**开启后**，当任何智能体试图提交一条被判为 **L2（危险）** 的命令时，Ridge 会**挂起那条命令、不让它执行**，并弹出一个居中的审批模态：

```
┌─────────────────────────────────────────────┐
│ 🛡️ 需要你的授权                         [L2]  │
│    来自 pane#2 (claude-code)                  │
│ ───────────────────────────────────────────  │
│ 递归/批量删除                                  │
│ ┌─────────────────────────────────────────┐ │
│ │ rm -rf /tmp/build-cache --no-preserve-root│ │
│ └─────────────────────────────────────────┘ │
│  ✕ 拒绝              ✏ 修改        ✓ 批准      │
└─────────────────────────────────────────────┘
```

你有三个选择：
- **✓ 批准**：放行，命令照常执行。
- **✕ 拒绝**：阻断，向智能体返回「被授权策略拦截」，它不会执行。
- **✏ 修改**：先改写命令（比如给路径加个限定），再执行你改写后的版本。

**特性**：
- 多条待审批会**排队**，模态右上角显示 `+N` 表示后面还有几条。
- 超时未裁决（默认 120 秒）会**自动拒绝**（fail-closed，宁可挡错不可放错）。
- **只拦 L2 危险命令**；L0/L1 直接放行，不打扰你。

> ⚠️ **默认关闭**。开启方式见 §7。关闭时智能体命令照常执行（行为与加这套系统之前完全一致）。

## 3. 风险分级（L0 / L1 / L2）

每条命令在执行前都会被分级，这是安全审批的依据：

| 档位 | 含义 | 例子 |
| --- | --- | --- |
| **L0** 只读 | 不改任何东西 | `ls -la`、`git status`、`cat x`、`grep ...` |
| **L1** 工作区写 | 在工作区内写文件 | `echo hi > out.txt`、`npm install`、改文件 |
| **L2** 危险 | 可能造成不可逆/外部影响 | `rm -rf ...`、`git push`、`curl ... | sh`、`sudo ...` |

分级器**抗绕过**：会先把多余空格归一化（`git   push` 照样判 L2）、识别管道灌 shell（`... | bash`）、重定向覆盖等花招，不会被简单变形骗过。

---

## 4. 循环熔断

AI 智能体偶尔会卡进「改代码 → 跑测试 → 同样报错 → 再改 → 还是同样报错」的死循环，白白烧 token。循环熔断就是给每个 Worker 装的**保险丝**。

**怎么触发**：当一个队员**连续 3 次**上报**高度相似的失败**（同一个错误特征），系统判定它陷入逻辑死锁，于是：
1. **向那个分屏发 `Ctrl+C`**，强制打断它正在做的事；
2. 在指挥部**置顶一条红色「熔断告警」**（见 §1），告诉你是谁、为什么；

你看到告警后接管那个队员（重新给指令、换个思路、或让 Leader 重新分派），然后点「清除」消掉告警。

> 熔断**默认随队工作**（无需手动开），但只有当队员通过协作 API 上报带「分屏 + 失败特征」的进度时才会计数。阈值（3 次）是内置的。

## 5. 智能体协作（TML / 派活 / 广播 / 汇报）

队员之间通过一套内部 HTTP API 协作，你**通常不用直接碰**——它们由智能体的运行时自动调用。你只需要在指挥部的「协作审计」里**看结果**。能力包括：

- **查兵马（team-profile）**：Leader 启动时拉取花名册，决定派给谁。
- **派活（delegate-task）**：Leader 把一个目标交给某个 Worker，会向那个分屏注入任务、把它标记为「工作中」，并返回一张任务票（Task Ticket）。
- **广播（broadcast）**：全网求助，「有谁能接这个活？」其他队员举手接单。
- **汇报（report-progress）**：Worker 把阶段性结果回报给 Leader（也是循环熔断的计数来源）。

这些动作都会以**人话**形式出现在「协作审计」栏里。

## 6. 内置 MCP server（高级 / 给开发者）

Ridge 自带一个标准 **MCP（Model Context Protocol）** 服务端，让 MCP-原生的智能体直接挂进来用 Ridge 的能力。

- **端点**：`ws://<teammate-host>/api/v1/mcp/ws`（teammate 服务的 WebSocket，带 bearer token 鉴权；地址/令牌通过分屏环境变量 `RIDGE_TEAMMATE_*` 注入）。
- **协议**：JSON-RPC 2.0，支持 `initialize` / `tools/list` / `tools/call` / `resources/read`。
- **工具**：`ridge_send_to_teammate`、`ridge_delegate_task`、`ridge_split_pane`、`ridge_stash_data`、`ridge_get_team_profile` 等。
- **资源**：`ridge://workspace/active-panes`（活动工作区花名册）等 `ridge://` URI。

普通用户不用关心这一层；它是给「自带 MCP 客户端的智能体」用的标准接入口。

## 7. 开关一览

这次新增的三个「网关」中，两个**默认关闭**（保证零行为变化、零回归），需要时再开。目前通过**命令开关**控制（设置面板里的可视化开关是后续计划）：

| 能力 | 默认 | 开关 | 说明 |
| --- | --- | --- | --- |
| **安全审批（HITL）** | 关 | `set_hitl_enabled(true/false)` | 开后才会对 L2 命令弹审批模态 |
| **TML 流净化 / 协作审计** | 关 | `set_tml_stream_enabled(true/false)` | 开后才会隐藏队员间的 TML 控制字符并把协作刷进「协作审计」 |
| **循环熔断** | 开（随队工作） | 无开关 | 队员上报失败达阈值即触发 |
| **数字指挥部面板** | 开（常驻侧栏） | 无开关 | 无团队时显示空态，不打扰 |

> 这两个默认关的开关之所以默认关，是因为它们触及命令执行 / 终端输出的关键路径——先确保对现有使用**零影响**，待你确认要用再逐个打开。开启它们目前需要由开发者/脚本调用对应命令；图形开关会在后续设置面板里补上。

## 8. 常见问题（FAQ）

**Q：我看不到「智能体指挥部」面板？**
A：① 确认你用的是包含本能力的新版本（需重新构建/安装）；② 确认左侧栏已展开、且当前有打开的工作区——它在侧栏的全局插件区（底部），跟着工作区一起出现。

**Q：花名册一直是空的 / 显示「还没有智能体加入」？**
A：只有当智能体通过 Ridge 的 teammate 机制**注册**到某个分屏后才会进花名册。你自己手动开的普通终端不算队员。

**Q：「协作审计」一直空？**
A：需要 ① 打开「TML 流净化」开关（§7），且 ② 队员之间真的在用 TML 通信。两者都满足才有内容。

**Q：我没看到危险命令的审批弹窗，它直接跑了？**
A：安全审批**默认关闭**，需先 `set_hitl_enabled(true)`（§7）。另外它**只拦 L2**，且只拦智能体经协作通道提交的命令——你自己在终端里手敲的命令不走这条审批闸。

**Q：熔断会误伤吗？**
A：只有「连续 3 次**高度相似**的失败」才触发；单次失败、或失败特征不同，都不会熔断。触发后是发 `Ctrl+C` 打断 + 告警，不会杀掉分屏。

**Q：这套东西会拖慢我平时用终端吗？**
A：不会。两个关键路径网关（审批、TML 净化）默认关时是**字节级零变化**；指挥部面板无团队时零渲染。
